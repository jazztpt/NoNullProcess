Code Reviews¶

Everyone¶

Guildelines¶

  • Accept that many programming decisions are opinions. Discuss tradeoffs, which you prefer, and reach a resolution quickly. [1]
  • Ask questions; don’t make demands. (“What do you think about naming this :user_id?”) [1]
  • Ask for clarification. (“I didn’t understand. Can you clarify?”) [1]
  • Avoid selective ownership of code. (“mine”, “not mine”, “yours”) [1]
  • Avoid using terms that could be seen as referring to personal traits. (“dumb”, “stupid”). Assume everyone is attractive, intelligent, and well-meaning. [1]
  • Be explicit. Remember people don’t always understand your intentions online. [1]
  • Be humble. (“I’m not sure - let’s look it up.”) [1]
  • Don’t use hyperbole. (“always”, “never”, “endlessly”, “nothing”) [1]
  • Don’t use sarcasm. [1]
  • Keep it real. If emoji, animated gifs, or humor aren’t you, don’t force them. If they are, use them with aplomb. [1]
  • Talk in person if there are too many “I didn’t understand” or “Alternative solution:” comments. Post a follow-up comment summarizing offline discussion. [1]

As the author¶

Guidelines¶

  • Be grateful for the reviewer’s suggestions. (“Good call. I’ll make that change.”) [1]
  • Don’t take it personally. The review is of the code, not you. [1]
  • Explain why the code exists. (“It’s like that because of these reasons. Would it be more clear if I rename this class/file/method/variable?”) [1]
  • Extract some changes and refactorings into future tickets/stories. [1]
  • Link to the code review from the ticket/story. (“Ready for review: https://github.com/organization/project/pull/1”) [1]
  • Push commits based on earlier rounds of feedback as isolated commits to the branch. Do not squash until the branch is ready to merge. Reviewers should be able to read individual updates based on their earlier feedback. [1]
  • Seek to understand the reviewer’s perspective. [1]
  • Try to respond to every comment. [1]
  • Wait to merge the branch until Continuous Integration (TDDium, TravisCI, etc.) tells you the test suite is green in the branch. [1]
  • Merge once you feel confident in the code and its impact on the project. [1]

Steps¶

  • [ ] Ensure that your branch can be merged cleanly
  • [ ] Ensure that all of the tests pass on your branch
  • [ ] Ensure that your code is documented [2]

As the reviewer¶

Guildelines¶

  • Communicate which ideas you feel strongly about and those you don’t.
  • Identify ways to simplify the code while still solving the problem.
  • If discussions turn too philosophical or academic, move the discussion offline to a regular Friday afternoon technique discussion. In the meantime, let the author make the final decision on alternative implementations.
  • Offer alternative implementations, but assume the author already considered them. (“What do you think about using a custom validator here?”)
  • Seek to understand the author’s perspective.
  • Sign off on the pull request with a :thumbsup: or “Ready to merge” comment.

Steps¶

  • [ ] Understand why the code is necessary [1]
  • [ ] Verify the pull request does what it says it does [2]
  • [ ] Verify test coverage and documentation [2]
  • [ ] Verify and run database migrations [2]
  • [ ] Verify code conventions, PEP8, PyChecker [2]
  • [ ] Run the application and validate front-end components [2]
